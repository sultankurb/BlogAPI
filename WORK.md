# ⚡ THE MANIFESTO: Прагматичный Модульный Монолит

Этот документ — фундамент проекта. Любое архитектурное решение, которое противоречит этим правилам, идет нахуй, потому что оно ломает производительность, усложняет поддержку и плодит технический долг. 

Мы пишем систему, которая должна летать, легко тестироваться и не падать под нагрузкой. Никаких иллюзий, только суровая реальность и личная выгода.

## 1. Базовая Философия
* **НАХУЙ МИКРОСЕРВИСЫ.** Проект строится как жесткий **Модульный Монолит**. Вся логика живет в одном процессе. Общение между доменами происходит в оперативной памяти (через вызовы функций/интерфейсы), а не по сети.
* **Слоистая архитектура:** Роутеры (FastAPI) -> Сервисы (Бизнес-логика) -> Репозитории (Работа с БД). 
* **Изоляция:** Роутер ничего не знает про SQL. Сервис ничего не знает про HTTP-запросы и токены. Репозиторий притворяется коллекцией в памяти.

## 2. Разделение на Домены
Проект распилен на независимые модули (домены). Домены не имеют права напрямую лезть в таблицы друг друга.

* **Identity Domain:** Отвечает за юзеров, авторизацию, роли и выдачу токенов.
* **Content Domain:** Отвечает за посты, комментарии и бизнес-логику платформы.

*Пример контракта:* Если `Content` хочет узнать, активен ли юзер, он не делает `JOIN` к таблице `users`, а дергает зависимость или метод сервиса из `Identity`.

## 3. Аутентификация и Роли (FastAPI + Redis)
* **Access Token (JWT):** Валидируется математически по SECRET_KEY (проверка подписи и времени жизни). 
* **Гибридный Blacklist (Мгновенный бан):** Для защиты системы от заблокированных деструктивных элементов, после математической валидации токена роутер делает быстрый точечный запрос в Redis (`GET blacklist:{user_id}`). Если юзер в бане — выдаем 403 мгновенно, не дожидаясь протухания JWT.
* **Refresh Token:** Живет долго. Хранится в Redis. 
* **Роли и Права:** Проверяются на уровне слоя Сервисов.
* **Управление сессиями:** В Redis хранится Set с активными рефреш-токенами юзера (`user_sessions:{user_id}`). 
* **Инвалидация и мусор:** Используется ленивая очистка. Если админ меняет права юзеру, новые права применяются в момент следующего рефреша (когда выдается новый access-токен). Мертвые токены удаляются из Set'а при логауте или при попытке рефреша сгоревшего токена.

## 4. База Данных (PostgreSQL + SQLAlchemy)
База — это самое узкое горлышко. Защищаем её любой ценой.

* **Никакого Auto-commit в Middleware.** Мидлварь или FastAPI `Depends` только *открывает* сессию БД. Управление транзакциями (`commit`, `rollback`) происходит ИСКЛЮЧИТЕЛЬНО внутри Сервисов (или через кастомный паттерн Unit of Work), когда вся бизнес-логика отработала без ошибок.
* **Индексы для Read-Heavy полей.** Для почты используется функциональный индекс `CREATE UNIQUE INDEX ... ON users (lower(email))`. 
* **Никаких N+1 запросов.** Все связи для отдачи на клиент подгружаются через жадную загрузку (`selectinload` или `joinedload`) прямо в методах репозитория.
* **Запрет на OFFSET для ленты.** Для пагинации тяжелых списков используется только Keyset Pagination (по курсору: `WHERE id < last_seen_id ORDER BY id DESC`).
* **Денормализация для скорости.** Счетчики (например, количество комментариев) хранятся отдельным полем в таблице поста и обновляются транзакционно, чтобы не делать `COUNT(*)` при каждом чтении.

## 5. Кеширование
* Тяжелые и часто читаемые данные (например, лента постов) дублируются в Redis. 
* При обновлении поста или добавлении комментария — ключ в Redis инвалидируется (удаляется), чтобы следующий запрос актуализировал кэш из Postgres.
